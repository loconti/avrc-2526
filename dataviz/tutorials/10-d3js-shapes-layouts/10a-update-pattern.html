10a — Update Pattern and selection.join

L9 finished with a chart that draws once. The first thing that turns a static chart into a living one is making it respond when the data updates — the user sorts, filters, scrubs a slider, switches a year. In 09b you used .join("circle") to draw a chart once; now we call the same code again with a different data array and let D3.js figure out the diff. Two new ideas make that work: key functions (so D3.js tracks each datum across calls) and custom enter / update / exit callbacks (so you can style each sub-selection differently). This is the foundation for every dynamic chart in the rest of L10.

How this tutorial is organised

Three blocks, in order:

  1. The two forms of the update pattern (§1-§2). The historical enter().merge(...) form first, then the modern selection.join shortcut that supersedes it.
  2. What changes when data is re-bound (§3-§4). Why a key function matters (a side-by-side index-vs-key demo on the same data), and how the three-callback form of .join lets you style enter / update / exit independently.
  3. The pattern in production (§5). A sortable bar chart of the world's most populous countries built up in two visible steps — first with an index join (the failure mode visible on a real chart), then with a key join.

1. The older explicit pattern

Before selection.join was added (D3.js v5.8, early 2019), every update wrote the three sub-selections by hand. You will still see this form in older examples:

// older pattern — three sub-selections, .merge() to combine enter+update
const sel = svg.selectAll("circle").data(data, d => d.id);

sel.exit().remove();                         // 1. remove old

sel.enter()                                   // 2. create new
    .append("circle")
    .attr("fill", "steelblue")
  .merge(sel)                                 // 3. enter + update together
    .attr("cx", d => x(d.value))
    .attr("cy", d => y(d.group))
    .attr("r", 5);

The pattern works but is verbose. selection.join is the modern shortcut.

2. .join() on the second call

The same .join("circle") you used in 09b does the right thing automatically when you re-bind it with a different array. Modern code rarely calls the explicit form above:

function render(data) {
    svg.selectAll("circle")
        .data(data, d => d.id)             // re-bind — D3.js diffs against the existing DOM
        .join("circle")                     // creates / keeps / removes to match
            .attr("cx", d => x(d.value))
            .attr("cy", d => y(d.group))
            .attr("r", 5);
}

render(initialData);
render(filteredData);                       // same join, new picture
render(sortedData);                         // again — every call diffs from the previous DOM

The single change vs. 09b: passing a second argument to .data() — the key function. Without it, the next section's reshuffle demo gives the wrong picture.

3. Keys preserve object identity

The second argument to .data() is the key function:

.data(data, d => d.id)

Without a key, D3.js matches data to elements by index. That is fine for a one-shot chart, but breaks the moment the data is sorted, filtered, or shuffled — D3.js thinks the first row always corresponds to the first DOM element, even after a sort.

Two charts of the same five items, one shuffle button. The only difference between them is the second argument to .data(): the left chart uses an index join (no key), the right uses a key join (d => d.id). Click Shuffle the data a few times and watch each circle's colour:

The trick is in how the colour is set. fill is assigned once at enter, using the datum bound at that moment. With the index join, the bound datum changes on every rebind but the existing element keeps its initial fill — so the colour stays welded to a slot. With the key join, each element keeps the same datum across rebinds, so its colour stays welded to the letter (the datum's identity).

4. Custom join behaviour — three callbacks

For richer behaviour, .join accepts three callbacks — one per sub-selection:

selection.data(data, d => d.id)
    .join(
        enter  => enter.append("circle")
            .attr("r", 0)                                          // start invisible
            .attr("fill", "#009E73")                               // green for new
            .call(en => en.transition().duration(400)
                .attr("r", 6)),                                    // grow into place

        update => update
            .attr("fill", "#0072B2"),                              // blue for kept

        exit   => exit
            .attr("fill", "#D55E00")
            .call(ex => ex.transition().duration(300)
                .attr("r", 0).remove())                            // shrink and go
    );

Each callback returns a selection. D3.js merges enter's return value with the update selection automatically, so subsequent .attr calls on the result of .join(...) run on both new and kept elements.

5. A sortable bar chart — the join in production

Top-15 most populous countries (UN 2023). Click the dropdown to sort by name or by population. We build it in two visible steps — first with an index join so the failure mode from §3 is unmistakable on a real chart, then with a key join that fixes it. 10b will pick up where step 2 leaves off and add the missing transition.

Step 1 — the static bar chart (no key, no transition)

Start from a plain horizontal bar chart. The bars are bound by index (no key function), and the sort dropdown calls a render function that re-sorts the data and updates the band scale. fill is set once on enter — exactly as in the §3 keys demo — so the colour is the visible identity carried by each DOM element. No transition: bars jump to their new y instantly:

function render() {
    data.sort(currentSort);
    y.domain(data.map(d => d.country));        // band scale follows the new order

    svg.selectAll("rect")
        .data(data)                              // ← no key: bound by INDEX
        .join(
            enter => enter.append("rect")
                .attr("x", 0)
                .attr("height", y.bandwidth())
                .attr("fill", d => colour(d.value))   // ← set ONCE on enter
                .attr("y",     d => y(d.country))
                .attr("width", d => x(d.value)),
            update => update,
            exit   => exit.remove()
        )
        .attr("y",     d => y(d.country))           // re-applied every render
        .attr("width", d => x(d.value));
}

Sort works — but watch the colours: each position keeps its colour, so the bar at row 1 keeps the dark-blue it inherited on the first render, even when the country at row 1 is now Bangladesh (small population — should be light blue). This is the index-join failure mode from §3 on a real chart:

Step 2 — add a key function

Single change: pass d => d.country as the second argument to .data(). Now each bar tracks its country across sorts — colours follow countries, not positions:

svg.selectAll("rect")
    .data(data, d => d.country)                 // ← KEY: each bar tracks its country
    .join("rect")
        .attr("y",      d => y(d.country))
        .attr("width",  d => x(d.value))
        .attr("height", y.bandwidth());

Sort again — the colours move with the bars now, but they jump instantly. We can read the result, just not the motion:

Step 2 gives the right result but the bars jump instantly. Following each country across the sort with the eye is hard. The missing ingredient is animation — and that is the whole subject of 10b.

Takeaway. Every D3.js chart that responds to change runs the same three steps: bind the new data with a key function, call .join, and let it create / keep / remove elements. For differentiated entry/exit, pass three callbacks. 10b picks up where Step 2 leaves off and adds transitions on top of the same join.
