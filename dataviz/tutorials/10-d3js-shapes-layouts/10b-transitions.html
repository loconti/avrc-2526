10b — Transitions

10a made the chart change; this tutorial makes the change visible over time. A transition interpolates a selection's attributes or styles between two states — bars resizing, circles moving, colours shifting. With one method call you can soften every diff produced by the update pattern; the bar race at the end of this tutorial reuses 10a's key-joined sortable bar pattern, wrapped in one 800 ms shared .transition() that animates bars, value labels, and axes against the same timeline.

How this tutorial is organised

Three blocks, in order:

  1. The mechanism (§1-§4). What a transition is, where it usually goes (right after a .join), how to chain transitions, and the three knobs — duration, delay, easing — that control its feel.
  2. Two patterns that compose well (§5-§6). A staggered transition built with a per-datum .delay, and the language of appearance / removal that falls out of combining transitions with the custom enter / exit callbacks from 10a.
  3. The bar race capstone (§7). Everything above on a real chart, with one shared transition driving bars, value labels, and axes against the same timeline.

1. What a transition does

The minimal call:

d3.select("#dot")
    .transition()
        .duration(750)
        .attr("cx", 300);

The selection's current state is read at the start; the target value is read from the call. D3.js produces ~60 intermediate frames per second, interpolating cx from its current value to 300 over 750 ms.

2. Transitions after a join

The most common place to attach a transition is right after a .join. The bound data has just been refreshed, so the join's "update" sub-selection now points at the new attribute targets:

// data is an array of { id, value, group } objects
svg.selectAll("circle")
    .data(data, d => d.id)               // key on a stable identifier
    .join("circle")
    .transition()
    .duration(600)
        .attr("cx", d => x(d.value))     // numeric → x position
        .attr("cy", d => y(d.group));    // categorical → y row

Read it as: bind the data, create or keep marks, then animate marks to their new attributes. Every D3.js chart that responds to change uses some variant of this pattern.

3. Chaining transitions

A transition can be followed by another transition. The second one starts when the first finishes:

d3.select("#dot")
    .transition().duration(600)
        .attr("cx", 200)            // 1. move right
    .transition().duration(600)
        .attr("cy", 100)            // 2. then move up
    .transition().duration(400)
        .attr("fill", "tomato");    // 3. then change colour

Useful for short, narrative steps where each phase has a clear meaning. Avoid long chains — viewers lose track of what changed when.

4. Tuning a transition: duration, delay, easing

Three knobs control the feel of every transition:

selection
    .transition()
        .duration(750)               // ms
        .delay(100)                  // ms before the transition starts
        .ease(d3.easeCubicInOut)     // shape of the time → progress curve
        .attr("cx", d => x(d.value));

D3.js ships about thirty easing functions. The five you'll reach for most often:

  • easeLinear — constant speed; mechanical feel
  • easeCubicInOut — accelerates and decelerates; the everyday default
  • easeBackOut — overshoots slightly then settles; "snappy"
  • easeBounceOut — bounces at the end; playful
  • easeElasticOut — wobbles into place; very playful

5. Staggered transitions with .delay

If .delay takes a function, each datum gets its own start time. Multiplying by the index (i) makes the elements animate in sequence — the eye follows the wave:

circles
    .transition()
    .duration(400)
    .delay((d, i) => i * 30)            // each one 30 ms later
        .attr("cx", d => x(d.value));

6. Transitions on enter / exit

Combined with the custom-join form from 10a, transitions become the language of appearance and removal:

selection.data(data, d => d.id)
    .join(
        enter  => enter.append("circle")
            .attr("r", 0)
            .call(en => en.transition().duration(400).attr("r", 6)),    // grow in
        update => update,
        exit   => exit.call(ex => ex.transition().duration(300)
            .attr("r", 0).remove())                                       // shrink out
    );

You already saw this idea in 10a §3 (the keys-vs-index demo, where shuffling the array reshuffled the marks). The composition is: data join → transition. That two-step phrase covers almost every animated chart in D3.js.

7. Bar race — transitions on a real chart

The Observable gallery's Bar chart race composes everything in this tutorial: a key-joined bar chart, a per-frame scaleBand.domain() that reorders the bars, and a 800 ms transition that animates each rect to its new y and width on every step.

One detail worth naming: bars, value labels, and axes all need to animate against the same timeline — if each one started its own transition with its own duration, you'd get small drifts (label arriving before its bar, axis lagging) that the eye reads as jitter. The fix is to define one transition and share it. Calling .transition(t) on a selection makes that selection inherit the existing transition t instead of starting a new one — same start time, same duration, same easing for every selection that asks for it:

function step(year) {
    const sorted = data.filter(d => d.year === year)
                       .sort((a, b) => d3.descending(a.value, b.value));
    y.domain(sorted.map(d => d.id));            // re-rank the band scale

    // ONE shared transition for the whole frame
    const t = svg.transition().duration(800).ease(d3.easeLinear);

    svg.selectAll("rect").data(sorted, d => d.id)
        .join("rect")
        .transition(t)                           // inherit the shared timeline
            .attr("y",     d => y(d.id))
            .attr("width", d => x(d.value));

    svg.selectAll("text.label").data(sorted, d => d.id)
        .join("text").attr("class", "label")
        .transition(t)                           // same timeline as the bars
            .attr("y", d => y(d.id) + y.bandwidth() / 2);

    xAxis.transition(t).call(d3.axisBottom(x));  // axes too
    yAxis.transition(t).call(d3.axisLeft(y));
}

// step through every year on a timer
let i = 0;
d3.interval(() => step(years[i++ % years.length]), 1200);

Why this matters in practice: a common bug is to write selection.call(T) where T is a transition-creating helper. That looks like it returns the transition, but selection.call always returns the selection — so subsequent .attr calls run synchronously and the chart snaps instead of animating. Inheriting via .transition(t) avoids the trap.

Takeaway. A transition is one method call, three knobs (duration, delay, easing), and one rule of taste: animate to explain change, not to perform. Combine transitions with the data join and you have everything needed to make charts feel alive.
