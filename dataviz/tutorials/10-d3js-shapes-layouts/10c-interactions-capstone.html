10c — Interactions

10a built change (the update pattern); 10b animated it (transitions). The missing piece is who triggers the change. In a static chart, nothing; in an interactive chart, the reader does — by hovering, clicking, scrubbing a slider. This tutorial closes the loop by wiring user events back into the same data join from 10a and the same transitions from 10b, on the same Gapminder scatterplot from 09e.

How this tutorial is organised

Three blocks, in order:

  1. The pattern (§1-§2). How D3.js forwards DOM events to your callbacks, and the three-step event → state → render loop that every interactive D3.js chart follows.
  2. Three interactions, one chart at a time (§3-§5). Hover (§3), click to toggle a selection (§4), and a year slider (§5) — each is built up step by step in isolation so you can read the diff between "static" and "interactive" cleanly.
  3. Capstone and helpers (§6-§8). One chart that combines hover + click + legend filter + slider; then a brief tour of D3.js's dedicated event helpers (d3.brush, d3.drag, d3.zoom); then a four-step walkthrough of d3.brush in detail.

1. Listening for user actions

D3.js selections forward DOM events through selection.on:

svg.selectAll("circle")
    .on("mouseover", (event, d) => {
        console.log("hover on", d.country);
    });

The callback gets two arguments:

  • event — the native browser event (with event.pageX, pageY, currentTarget, etc.)
  • d — the datum bound to the element that fired the event

This is how visual marks become interactive. The most common events:

  • mouseover · mouseout · mousemove
  • click
  • input · change (form controls)
  • keyboard events on the document or window

2. The event → state → render loop

Most interactive D3.js charts follow the same three-step loop:

  1. Event — a user action fires (a hover, a click, a slider drag).
  2. State — the callback updates a small piece of state (a selected continent, a current year).
  3. Render — a single render function reads the state and re-runs the data join.
let selectedContinent = null;       // state

function render() {                  // render reads state
    const visible = selectedContinent
        ? data.filter(d => d.continent === selectedContinent)
        : data;

    svg.selectAll("circle")
        .data(visible, d => d.country)
        .join("circle")
            .attr("cx", d => x(d.gdpPercap))
            .attr("cy", d => y(d.lifeExp))
            ...;
}

legend.on("click", (event, d) => {   // event updates state
    selectedContinent = (selectedContinent === d) ? null : d;
    render();
});

This split keeps event callbacks tiny — they only touch state. All drawing logic lives in one place, which makes the chart easier to debug and extend.

3. Hover tooltip — the simplest interaction

The smallest useful interaction: a tooltip on hover. We use a single absolutely-positioned <div> as a shared tooltip rather than appending one DOM element per row.

const tip = d3.select("#tooltip");

svg.selectAll("circle")
    .on("mouseover", (event, d) => {
        tip.style("opacity", 1)
           .html(`<b>${d.country}</b><br>life exp: ${d.lifeExp.toFixed(1)} yr`);
    })
    .on("mousemove", (event) => {
        tip.style("left", (event.pageX + 12) + "px")
           .style("top",  (event.pageY + 12) + "px");
    })
    .on("mouseout", () => tip.style("opacity", 0));

4. Click to toggle — state lives in a variable

The simplest interaction beyond hover is a toggle: click a mark, it lights up; click it again, it goes back. We will build a country selector. Click any circle and it becomes the selected country (thick black ring, every other circle fades into the background); click the same one again, or click the empty area, and the selection clears.

Step 1 — declare the state

The whole interaction is about which country, if any, is currently selected. That is one piece of information, so it gets one variable. We use null for "nothing selected":

let selected = null;        // either a country name (e.g. "Italy") or null

Notice this lives outside any handler — it has to survive between events. Every click will read it, change it, and read it again on the next click.

Step 2 — write a render function that reads the state

The render function does only one thing: it looks at selected and decides how each circle should be drawn. We use the data join from 10a with a key on d.country so each circle stays bound to its country across renders:

function render() {
    svg.selectAll("circle")
        .data(data, d => d.country)
        .join("circle")
            // 1. positional attributes — same on every render
            .attr("cx", d => x(d.gdpPercap))
            .attr("cy", d => y(d.lifeExp))
            .attr("r",  d => r(d.pop))
            // 2. attributes that depend on the SELECTION
            .attr("stroke",       d => d.country === selected ? "#1a1a1a" : "white")
            .attr("stroke-width", d => d.country === selected ? 3 : 1)
            .attr("opacity",      d => selected && d.country !== selected ? 0.15 : 0.85);
}

Look at the three "depend on selection" lines:

  • stroke — black if this circle's country is the selected one, white otherwise.
  • stroke-width — fat (3) for the selected circle, thin (1) for the rest.
  • opacity — when something is selected, dim every other circle to 0.15; when nothing is selected (selected === null), the selected && … short-circuits to null (falsy), so every circle stays at the normal 0.85.

So if selected is null, render produces a plain scatter. If selected is "Italy", render produces the same scatter with Italy decorated and the rest dimmed.

Step 3 — wire the click to update the state and re-render

The click handler is two lines. It updates selected with the toggle rule, then calls render() so the visuals catch up:

.on("click", (event, d) => {
    selected = (selected === d.country) ? null : d.country;   // toggle
    render();                                                  // redraw
});

Read the toggle line as a question: "is this country already the selected one?" — if yes, set selected back to null (deselect); if no, make this country the new selection.

Mounting the click handler inside render means every circle ends up with the handler — newly entered circles get it on creation, existing ones get it re-attached every render (a no-op replace). The handler is one short closure that captures selected and render from the surrounding scope.

Step 4 — let the background click clear the selection

One small affordance: if the user clicks anywhere on the chart that isn't a circle, treat it as "deselect". We attach a click handler to the SVG itself and check event.target:

svg.on("click", (event) => {
    if (event.target.tagName !== "circle") {
        selected = null;
        render();
    }
});

SVG events bubble up the same way HTML events do, so a click on a circle reaches both handlers. The circle handler runs first; the SVG handler then sees event.target.tagName === "circle" and ignores the event. A click on the empty background reaches only the SVG handler, where the tag check fails and the selection is cleared.

Putting it together

Three rules summarise the whole pattern:

  1. One variable holds the state. If you find yourself adding a second to track the same thing, you usually want one combined object instead.
  2. Event handlers only update state, then call render. They never paint pixels themselves.
  3. render is the only place that reads state and produces visuals. Every visual difference between "before click" and "after click" must be expressible inside render as a function of the state variables.

This is exactly the loop you will reuse for sliders (§5), brushes (§8), and the L11 force layouts. Try it: click any country, click another to switch, click the same one again — or click the background — to clear:

5. The year slider — scrubbing through time

Gapminder spans the years 1952–2007. A range slider lets the reader move along that timeline; each tick of the slider updates a piece of state (the current year) and re-renders the chart with that year's data. The structure is the same loop as §4, but the state now holds a year instead of a selected country, and circles slide between positions instead of just changing colour.

Step 1 — declare the state

One variable holds the year currently shown on the chart:

let currentYear = 1952;     // the year the chart is drawing right now

Step 2 — render reads the state and filters the data

The render function looks at currentYear, picks the rows for that year, and joins them to circles with a key on d.country so each country keeps the same DOM element across years:

function render() {
    const yearData = data.filter(d => d.year === currentYear);

    svg.selectAll("circle")
        .data(yearData, d => d.country)         // ← key join from 10a
        .join(
            enter  => enter.append("circle")
                .attr("cx", d => x(d.gdpPercap))
                .attr("cy", d => y(d.lifeExp))
                .attr("r",  d => r(d.pop))
                .attr("fill", d => colorOf(d.continent)),
            update => update,                    // existing circles are kept
            exit   => exit.remove()              // a country with no row this year goes away
        );
}

Why a key join? Without one, the circle at index 1 in 1952 would be reused for whichever country happens to land at index 1 in 1957 — circles would visually swap identities every tick. With d => d.country, "Italy"'s DOM circle stays bound to Italy for the whole 1952→2007 span; only its cx / cy / r change.

Step 3 — wire the slider's input event

HTML <input type="range"> fires an input event on every drag position (continuous), and a change event when the user releases (final value only). For scrubbing we want continuous updates, so we listen to input:

const yearSlider = d3.select("#year");

yearSlider.on("input", function () {
    currentYear = +this.value;     // string → number
    render();                       // redraw with the new state
});

Same shape as the click handler in §4: update the state variable, then call render. The + in front of this.value coerces the slider's string value to a number — without it the filter d.year === currentYear would always be false (string vs number).

Step 4 — animate cx, cy, r with a transition

If we stop here, every slider tick would snap circles to their new positions. That is correct but jarring: the eye loses track of which country is which. Wrapping the position attributes in a .transition() (from 10b) makes the circles glide between consecutive years:

const t = svg.transition()
    .duration(450)
    .ease(d3.easeCubicInOut);   // shared 450 ms timeline for the whole frame

svg.selectAll("circle")
    .data(yearData, d => d.country)
    .join(
        enter  => enter.append("circle")
            .attr("cx", d => x(d.gdpPercap))
            .attr("cy", d => y(d.lifeExp))
            .attr("r",  d => r(d.pop))
            .attr("fill", d => colorOf(d.continent)),
        update => update,
        exit   => exit.remove()
    )
    .transition(t)                              // inherit the shared timeline
        .attr("cx", d => x(d.gdpPercap))
        .attr("cy", d => y(d.lifeExp))
        .attr("r",  d => r(d.pop));

Two subtle points worth naming:

  • Enter sets attributes instantly, then the merged selection transitions. New circles appear in the right place on year 1; from then on they slide.
  • Drag fast and the transitions interrupt cleanly. If a new input event arrives mid-flight, D3.js cancels the in-flight transition and starts a new one from the circle's current position — no jitter, no jumps.

7. Built-in interaction helpers

For richer behaviours D3.js ships dedicated helpers that handle the messy event details so your callback stays declarative:

  • d3.brush — drag a rectangular selection over the chart (filtering, range queries)
  • d3.drag — make individual marks draggable (used heavily in force-directed graphs in L11)
  • d3.zoom — pan and zoom transforms on a chart (covered in 10e)

Each one follows the event → state → render pattern: the helper fires events, your callback updates state, a render function redraws. Same shape, different inputs.

8. d3.brush — selecting a range

A brush is a draggable rectangular selection that the user paints onto the chart. Inside the rectangle: the data points the reader cares about. Outside: everything else, which we usually fade. The reader can resize the brush from its handles, drag it wholesale to translate it, and click the empty area to clear it. D3.js ships three brush helpers — d3.brushX (selection along x only), d3.brushY (along y only), and d3.brush (both axes). They follow the same event → state → render loop you have used in §4-§5; the only new ingredient is that the event source is now a D3.js helper instead of a single DOM listener.

Step 1 — create a brush behaviour

A brush is a behaviour object: you configure it, then apply it to an SVG group. Two settings matter:

const brush = d3.brushX()
    .extent([[0, 0], [innerW, innerH]])     // pixel rectangle the brush is allowed to live in
    .on("brush end", handler);              // events to listen to

extent is the rectangle (in pixel coordinates of the inner plot area) inside which the brush is allowed to be drawn. The two events worth listening to are:

  • brush — fires continuously while the user is dragging (every pixel of motion). Use this for live updates.
  • end — fires once when the user releases the mouse, and once with selection === null when the user clicks outside to clear. Use this to detect "the brush has been emptied".

Listening to both ("brush end") gives you live highlighting plus a clean reset on click.

Step 2 — write the handler: pixel → data, then re-style the marks

The brush event object carries a selection field — either an array [x0, x1] in pixels (for brushX) or null (when the brush has been cleared). We invert the pixels through the x scale to get GDP values, then set each circle's opacity according to whether its GDP falls inside the band:

function handler({ selection }) {
    if (!selection) {                                    // brush cleared
        dots.attr("opacity", 0.7);                       //   → restore default opacity
        return;
    }
    const [x0, x1] = selection.map(x.invert);            // pixel → GDP per capita
    dots.attr("opacity", d =>
        (d.gdpPercap >= x0 && d.gdpPercap <= x1) ? 1 : 0.1);
}

Three lines of decisions:

  • The early return handles the "click outside to clear" case. Without it, the next line would crash because null.map(...) is an error.
  • selection.map(x.invert) uses the scale's inverse function to translate pixel positions back into data values. This is why scales matter so much in D3.js — they are bidirectional.
  • The opacity ternary is a tiny render: each circle's opacity is a function of one piece of state (the inverted band [x0, x1]). Same shape as §4's selected ternary.

Step 3 — attach the brush to a group on top of the marks

The brush renders as a small group of SVG elements (an overlay rectangle, the selection rectangle, and resize handles). It must sit on top of the marks so it can intercept mouse events:

svg.append("g")
    .attr("class", "brush")
    .call(brush);              // applies the configured behaviour to the group

selection.call(brush) is the standard D3.js idiom for invoking a behaviour on a selection — the same pattern d3.zoom and d3.drag use in 10e and 11c. Once you have called it, the user can paint rectangles by clicking-and-dragging.

Same shape as the click / slider handlers in §4-§5: an event fires, a piece of state (here the inverted GDP range [x0, x1]) is updated, marks are re-attributed. Linked views — the canonical D3.js dashboard pattern, where painting on one chart highlights related rows in others — are nothing more than this same handler updating a piece of state shared across several charts. Each chart has its own render; all of them read the same selection.

Takeaway. Three event sources (selection.on, d3.brush, sliders) feeding one render function via a shared piece of state. From here, every chart in the L11 / L12 unit (force layouts, chord diagrams, sankey) is a richer composition of the same primitives — and L11's drag interaction is just d3.drag wired to fx/fy, exactly the helper pattern shown above.
