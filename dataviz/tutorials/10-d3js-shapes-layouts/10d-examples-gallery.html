10d — Layouts: bar, stacked bar, streamgraph

10a, 10b, and 10c built the three L10 mechanisms — update pattern, transitions, and events. This tutorial spends them on real chart types from the D3.js Observable gallery: a bar chart (the simplest layout-driven chart), a stacked bar chart (introducing the d3.stack layout — a function that turns a flat row-oriented array into per-segment [y0, y1] intervals), and a streamgraph (same stack, different offset and curve). Closes with a which layout for which question? decision table — picking the right layout is half the job.

How this tutorial is organised

Three chart types, each built up in visible stages, then a decision aid:

  1. Bar chart (§1-§3). One <rect> per category. We build it in five steps — empty canvas → x positions → y heights → axes → polish — and then look at the scaleBand padding knobs that control the gaps between bars.
  2. Stacked bar chart (§4-§6). Same bars, multiple series stacked on top of each other. We motivate the new ingredient (we need running totals — flat arrays don't carry them), introduce d3.stack() as the helper that produces them, and build the chart in three steps using a nested join.
  3. Streamgraph (§7-§8). Same d3.stack() output drawn as flowing areas instead of rectangles, with a different offset (the baseline shape) and a smoothing curve. Three steps from flat baseline → centred → canonical wiggle, plus an interactive playground.
  4. Choosing a layout (§9-§10). One short paragraph on what makes d3.stack a "layout" in the D3.js sense, then a decision table covering every layout you will meet in L11 / L12.

1. A simple bar chart

The simplest layout-driven chart. A bar is a <rect> with four attributes:

  • x — left edge of the bar
  • y — top edge of the bar (not the bottom!)
  • width — bar width
  • height — bar height

Because SVG y grows downward, the top of a bar has the smaller y. The height is then computed as the distance from the top down to the baseline. Most bar-chart bugs in D3.js trace back to this single fact.

2. Bar chart construction, step by step

Mean life expectancy per continent for 2007 — five values, five bars. We build it in five visible stages, the same way 09e built the scatterplot.

Step 1 — the data and an empty canvas

Start from the data and the standard margin convention. Nothing drawn yet — the canvas is a rectangle of known inner dimensions innerW × innerH we can fill in the next steps:

const barData = [
    { continent: "Africa",   mean: 54.8 },
    { continent: "Americas", mean: 73.6 },
    { continent: "Asia",     mean: 70.7 },
    { continent: "Europe",   mean: 77.6 },
    { continent: "Oceania",  mean: 80.7 }
];

const margin = { top: 20, right: 20, bottom: 40, left: 40 };
const innerW = width  - margin.left - margin.right;
const innerH = height - margin.top  - margin.bottom;

const svg = d3.select("#chart").append("svg")
    .attr("width", width).attr("height", height);
const plot = svg.append("g")
    .attr("transform", `translate(${margin.left}, ${margin.top})`);

Step 2 — place the bars horizontally with scaleBand

One slot per continent. With fixed heights for now, just to confirm the horizontal positions are right:

const x = d3.scaleBand()
    .domain(barData.map(d => d.continent))
    .range([0, innerW])
    .padding(0.15);

plot.selectAll("rect").data(barData).join("rect")
    .attr("x",      d => x(d.continent))
    .attr("y",      0)
    .attr("width",  x.bandwidth())
    .attr("height", innerH)               // ← placeholder: full plot height
    .attr("fill",   "#0072B2").attr("opacity", 0.4);

Step 3 — encode the value as height with scaleLinear

Now the bars reflect the data. Note the inverted range ([innerH, 0]) so larger values appear higher, and the height formula innerH - y(d.mean) grows the bar upward from the baseline:

const y = d3.scaleLinear()
    .domain([0, d3.max(barData, d => d.mean)]).nice()
    .range([innerH, 0]);                  // inverted on purpose

plot.selectAll("rect").data(barData).join("rect")
    .attr("x",      d => x(d.continent))
    .attr("y",      d => y(d.mean))       // top of the bar
    .attr("width",  x.bandwidth())
    .attr("height", d => innerH - y(d.mean));   // top → baseline

If you swap height = y(d.mean) by mistake, every bar hangs from the top of the chart (try it locally — the result is unmistakable).

Step 4 — axes

Two axis generators, one per scale, draw ticks and labels at the edges of the plot:

plot.append("g").attr("class", "axis")
    .attr("transform", `translate(0, ${innerH})`)
    .call(d3.axisBottom(x));

plot.append("g").attr("class", "axis")
    .call(d3.axisLeft(y));

Step 5 — the finished bar chart

Same code, with a per-continent fill from the shared palette, a touch of opacity, and a value label above each bar. This is what the rest of the tutorial assumes when it says "the bar chart":

plot.selectAll("rect.bar").data(barData).join("rect")
    .attr("class", "bar")
    .attr("x",      d => x(d.continent))
    .attr("y",      d => y(d.mean))
    .attr("width",  x.bandwidth())
    .attr("height", d => innerH - y(d.mean))
    .attr("fill",   d => PALETTE[d.continent])
    .attr("opacity", 0.85);

plot.selectAll("text.value").data(barData).join("text")
    .attr("class", "value")
    .attr("x", d => x(d.continent) + x.bandwidth() / 2)
    .attr("y", d => y(d.mean) - 6)
    .attr("text-anchor", "middle")
    .text(d => d3.format(".1f")(d.mean));

3. The padding knobs on scaleBand

You met scaleBand in 09c §7 — x("category") for the slot's left edge, x.bandwidth() for the slot width. The piece you didn't need there is padding, which controls the gap between bars and the gap between bars and the plot edges:

d3.scaleBand()
    .domain([...])
    .range([0, innerW])
    .paddingInner(0.15)     // gap BETWEEN bars (fraction of step)
    .paddingOuter(0.25)     // gap at each end of the range
    .align(0.5);            // where the leftover space goes (0 = left, 1 = right)

// shorthand:
.padding(0.15)              // sets paddingInner AND paddingOuter to 0.15

Below: same seven categories, three different inner paddings. Padding controls the gap between bars; pick a value that lets the bars dominate without crowding.

4. Why stacked bars need transformed data

The plain bar chart had one number per x-category — easy: one <rect>, top edge at y(d.value). A stacked bar chart shows several categories piled on top of each other at the same x-position (apples + bananas + cherries per year). The visible shape now requires more information per segment than the input row directly carries:

  • the x-category (which bar)
  • the lower y of the segment (where it starts)
  • the upper y of the segment (where it ends)
  • the series key (which colour)

The lower and upper edges are running totals — the cherries layer's lower edge is apples + bananas, its upper edge is apples + bananas + cherries. Computing them by hand every render would be tedious and error-prone, so D3.js ships a helper that does the arithmetic and packages the result in a join-friendly shape: d3.stack().

5. d3.stack() transforms columns into layers

Input: an array of "wide" rows where each row is one x-position and each column is one series.

const data = [
    { year: 2018, apples: 10, bananas: 20, cherries:  5 },
    { year: 2019, apples: 12, bananas: 18, cherries:  8 },
    { year: 2020, apples: 15, bananas: 22, cherries:  6 },
    { year: 2021, apples: 17, bananas: 24, cherries: 10 },
    { year: 2022, apples: 14, bananas: 21, cherries: 12 }
];

const stack = d3.stack().keys(["apples", "bananas", "cherries"]);
const series = stack(data);

Output: an array of layers, one per key. Each layer has one entry per x-position, and each entry is a [y0, y1] pair (the bottom and top of the segment). The original row is attached as .data:

// series[0]  → the "apples" layer
// series[0][0] → [0, 10]   for year 2018  (apples stacked first, so y0 = 0)
// series[1][0] → [10, 30]  for year 2018  (bananas on top of apples)
// series[2][0] → [30, 35]  for year 2018  (cherries on top of bananas)
// series[0][0].data → { year: 2018, apples: 10, bananas: 20, cherries: 5 }

6. Drawing a stacked bar chart, step by step

The output of d3.stack() (an array of layers, each holding per-x segments) maps directly onto a nested data join: one <g> per layer, one <rect> per segment inside it. Below is the full pattern; the next three sub-sections build it up one layer at a time.

svg.selectAll("g.layer")
    .data(series)
    .join("g")
        .attr("class", "layer")
        .attr("fill", d => color(d.key))    // colour set ONCE per layer
  .selectAll("rect")
    .data(d => d)                            // children = segments of THIS layer
    .join("rect")
        .attr("x",      d => x(d.data.year))
        .attr("y",      d => y(d[1]))        // top of the segment
        .attr("height", d => y(d[0]) - y(d[1]))   // top − bottom
        .attr("width",  x.bandwidth());

The y-scale's domain is the maximum upper edge across all layers, not the max of any single column — because what we are plotting is the running total, not the individual values:

const yMax = d3.max(series, layer => d3.max(layer, d => d[1]));
const y = d3.scaleLinear().domain([0, yMax]).nice().range([innerH, 0]);

Step 1 — draw a single layer first

Before nesting the join, draw just the "apples" layer (series[0]) as a flat data join. This shows what one layer looks like in isolation: each segment's y is its upper edge d[1], and its height is y(d[0]) − y(d[1]). Axes are included from the start so the bars have a visible frame of reference:

plot.append("g").attr("class", "axis")
    .attr("transform", `translate(0, ${innerH})`)
    .call(d3.axisBottom(x).tickFormat(d3.format("d")));
plot.append("g").attr("class", "axis").call(d3.axisLeft(y));

plot.selectAll("rect").data(series[0]).join("rect")
    .attr("x",      d => x(d.data.year))
    .attr("y",      d => y(d[1]))
    .attr("width",  x.bandwidth())
    .attr("height", d => y(d[0]) - y(d[1]))
    .attr("fill",   color("apples"));

Step 2 — stack all three layers via the nested join

Now bind series (the array of all three layers) to <g> elements, and inside each <g> bind that layer's segments to <rect> children. The outer fill on each <g> propagates to its child <rect>s, so the colour is set in one place per layer. Axes carry over from step 1; only the legend is still missing:

plot.selectAll("g.layer").data(series).join("g")
        .attr("class", "layer")
        .attr("fill", d => color(d.key))           // colour set per LAYER
    .selectAll("rect").data(d => d).join("rect")    // children = segments of THIS layer
        .attr("x",      d => x(d.data.year))
        .attr("y",      d => y(d[1]))
        .attr("width",  x.bandwidth())
        .attr("height", d => y(d[0]) - y(d[1]));

Step 3 — add axes, the legend, and styling

Same nested join, plus axis generators at the edges of the plot and a categorical legend above the chart (the shared legend() helper from _shared/chart.js populates an empty <div class="legend"></div> with one swatch + label per scale entry):

// in the page: <div class="legend" id="legend-stack"></div>
legend("#legend-stack", color);

plot.append("g").attr("class", "axis")
    .attr("transform", `translate(0, ${innerH})`)
    .call(d3.axisBottom(x).tickFormat(d3.format("d")));
plot.append("g").attr("class", "axis").call(d3.axisLeft(y));

// the same nested join from Step 2, with .attr("opacity", 0.9) added.

7. Streamgraph — the same stack, drawn as flowing areas

The stacked bar chart in §6 used three ingredients: d3.stack() output, a scaleBand on the x-axis, and one <rect> per segment. Replace the rectangles with d3.area() paths and switch the scaleBand for a continuous scaleLinear on x and you get a streamgraph: the same per-segment [y0, y1] intervals from d3.stack(), drawn as smooth flowing ribbons instead of discrete bars. The two controls that turn a stacked area into a "real" streamgraph are the stack's offset (the shape of the baseline — flat / centred / wiggle-minimised) and the area's curve (sharp segments vs. smooth interpolation).

The dataset matters. A streamgraph only earns its keep when series rise and fall at different times — that's what stackOffsetWiggle is for. Monotonic data (every series only going up, like population over decades) makes wiggle / silhouette / none look nearly identical, defeating the chart. Mike Bostock's classic streamgraph gist and Byron & Wattenberg's original paper both use synthetic random-bump data for exactly this reason: 12 series × 200 time points where each series is a sum of random Gaussian bumps. We use the same construction below — the patterns generalise to any real dataset of the same shape (BLS unemployment by industry, music genres over time, etc.).

Three visible stages on the same synthetic data:

Step 1 — stacked area with the default offset

d3.stackOffsetNone uses a flat baseline at y = 0 — the result is a stacked area with the same flat baseline as §6's stacked bar, just drawn with d3.area() ribbons instead of <rect>s. The data shape and the join are identical to §6; only the geometry changes:

const stack = d3.stack().keys(seriesKeys);    // default: stackOffsetNone

const area = d3.area()
    .x(d => x(d.data.t))
    .y0(d => y(d[0]))
    .y1(d => y(d[1]));                          // straight segments, no curve yet

svg.selectAll("path.layer").data(stack(rows)).join("path")
    .attr("d", area)
    .attr("fill", d => colour(d.key));

Step 2 — switch to a centred offset

d3.stackOffsetSilhouette reflects the stack around zero so it grows symmetrically from a horizontal centre line. The total height per year is still the sum of all continents — it just sits half above and half below the axis:

d3.stack().keys(seriesKeys)
    .offset(d3.stackOffsetSilhouette);          // ← centred

Step 3 — minimise wiggle and add a curve

d3.stackOffsetWiggle (Byron & Wattenberg, 2008) chooses the baseline that minimises the slope of every layer — so the eye follows each stream more easily. Combine it with d3.stackOrderInsideOut (large layers in the middle) and a smooth curveBasis in the area generator and you get the canonical streamgraph look:

const stack = d3.stack().keys(seriesKeys)
    .offset(d3.stackOffsetWiggle)
    .order(d3.stackOrderInsideOut);

const area = d3.area()
    .x(d => x(d.data.t))
    .y0(d => y(d[0]))
    .y1(d => y(d[1]))
    .curve(d3.curveBasis);                      // smooth interpolation

8. Streamgraph — interactive playground

D3.js ships a small zoo of stack offsets and orders. The interactive version below lets you mix and match — same data, same layout, very different pictures (the Observable gallery's Streamgraph notebook does the same).

  • stackOffsetNone — flat baseline (default; stacked-bar shape)
  • stackOffsetExpand — every column normalises to 1 (100% stacked)
  • stackOffsetSilhouette — centred around zero (symmetric streamgraph)
  • stackOffsetWiggle — minimises layer slope (Byron & Wattenberg)

9. d3.stack is one example of a D3.js layout

Step back for a second. d3.stack() is a function that took flat input rows and produced drawable geometry — per-segment [y0, y1] intervals. That contract — data goes in, drawable geometry comes out — is what D3.js means by a layout. There is a small family of them, and their API shapes are all recognisably similar:

  • d3.stack — stacked bars / areas (this tutorial).
  • d3.pie — angles for pie / donut slices. Returns one { startAngle, endAngle, value, ... } per row.
  • d3.hierarchy + d3.tree / d3.treemap / d3.pack — hierarchical layouts. Take a tree, add x/y or rectangles or radii to every node.
  • d3.forceSimulation — physical simulation for force-directed graphs (L11). Mutates each node's x / y tick by tick.
  • d3.chord + d3.ribbon — pair-wise weighted relationships laid out around a circle (L12).

Once you've built one chart with d3.stack, the rest of the family is approachable: feed in the data, read out the geometry, hand the geometry to a data join. The mechanical pattern is identical; only the geometry changes.

10. Which layout for which question?

The hard part of a layout-driven chart isn't writing the code — every d3.* layout follows the same shape — but choosing the right one for the analytical question. A short decision table, including the layouts you'll meet in L11 / L12:

Question / data shapeLayoutWhy
One measure across categories ("mean life expectancy by continent") scaleBand + plain <rect> No layout transformation needed — bar chart, simplest case.
Sum-to-total comparison over a categorical x-axis d3.stack().offset(stackOffsetNone) Stacked bars / areas with a flat baseline. Absolute totals legible.
Share comparison ("what fraction of each year is apples?") d3.stack().offset(stackOffsetExpand) Each column normalises to 1 — 100% stacked. Parts-of-whole over time.
Trend visibility for many series at once ("how does each genre rise and fall over decades?") d3.stack().offset(stackOffsetWiggle) (streamgraph) Centred ribbon minimises layer slope (Byron & Wattenberg). Easier to follow individual streams than a flat-baseline stack.
One categorical breakdown at one moment ("market share now") d3.pie + d3.arc Donut / pie. Avoid for > 6 categories or close-in values.
Hierarchy with quantitative weight ("disk usage by directory") d3.hierarchy + d3.treemap / d3.partition / d3.pack Recursive subdivision. Treemap fills a rectangle, partition makes sunbursts, pack makes circle-packing.
Connections without a hierarchy ("character co-occurrence") d3.forceSimulation Physics-driven layout — see L11.
Pair-wise weighted relationships in a small set d3.chord + d3.ribbon Matrix bent into a circle — see L12 §12b.
Directed flow between layers ("budget by department by year") d3.sankey (plugin) Flow diagram — see L12 §12c.

The mechanical pattern is identical across every D3.js layout: data goes in, drawable geometry comes out. The skill is matching the geometry to the question.

Takeaway. 10d shows the three L10 mechanisms (update / transition / event) applied to layout-driven chart types — bar, stacked bar, streamgraph. The layouts themselves come from the same family (d3.stack here; d3.pie, d3.hierarchy, d3.forceSimulation elsewhere) and obey the same "data in, drawable geometry out" contract. Picking the right layout for the question is half the job — see the decision table above. 10e adds the navigation primitive (d3.zoom) so any of these charts can hold more data than fits on one screen.
