10e — Zoom and Pan

10a–10c made the chart respond to change, time, and input; 10d added composition via layouts. The piece left is what happens when the chart has more data than fits comfortably on the screen. The reader needs to zoom in on a region and pan around. d3.zoom attaches the standard zoom + pan affordances (mouse wheel, drag, double-click, pinch on touch) to a target element and emits a transform on every change. We build the canonical zoomable scatterplot in four visible stages on the gapminder 2007 slice from 09e.

How this tutorial is organised

Three blocks, in order:

  1. What d3.zoom is (§1). A short tour of the behaviour object: how it is configured, what events it fires, what an event transform looks like.
  2. Build a zoomable scatterplot in four visible steps (§2-§5). Static chart → naïve group transform → rescale-the-scales → clip to the plot rectangle. Each step adds one idea on top of the previous so the reason for every line is visible.
  3. When to use which pattern, plus a production checklist (§6-§7). Group-transform vs rescale-scales (the two valid recipes for different chart families), then the things you actually need to think about before shipping — responsive sizing, accessibility, SVG vs canvas, animation honesty.

1. d3.zoom in one minute

Like d3.brush from 10c §8, d3.zoom is a behaviour object: you configure it, then apply it to an SVG element with selection.call(zoom). The behaviour wires up wheel, drag, double-click, and touch pinch listeners on that element, and on every change it fires a "zoom" event. Your job is to read the event's transform and translate it into visual updates:

const zoom = d3.zoom()
    .scaleExtent([1, 16])                           // zoom range: 1× to 16×
    .extent([[0, 0], [innerW, innerH]])             // where in the SVG the zoom applies
    .translateExtent([[0, 0], [innerW, innerH]])    // pan stops at the plot edges
    .on("zoom", (event) => {
        const t = event.transform;                  // a {k, x, y} object
        // … apply t to your marks (next sections show how)
    });

svg.call(zoom);                                     // attach to the SVG

Four concepts to name now so the next four sections read smoothly:

  • scaleExtent([min, max]) — the zoom multiplier range. [1, 16] means the user can zoom up to 16× in but no further out than the initial 1×. Forgetting to set this and your reader can scroll-wheel themselves into sub-pixel land.
  • extent([[x0, y0], [x1, y1]]) — the pixel rectangle of the SVG that the zoom captures gestures from. Usually this is the inner plot area.
  • translateExtent([[x0, y0], [x1, y1]]) — the rectangle the user is allowed to pan to. Without it, the reader can drag the chart entirely off-screen and lose it. Set it equal to extent for the obvious-correct behaviour: pan stops at the edges of the plot.
  • The transform object — three numbers: k (zoom multiplier, ≥ 1 with our scale-extent), x and y (translation in pixels). Together they describe an affine transform. You'll see two ways to consume it below: applying it directly as an SVG transform attribute (§3), or using its .rescaleX / .rescaleY methods to rebuild the scales (§4-§5).

The gestures the behaviour wires up automatically — try them on every demo from §3 onward:

  • Mouse wheel (or trackpad scroll) — zoom in / out, centred on the cursor.
  • Click and drag — pan.
  • Double-click — zoom in 2×, centred on the click.
  • Shift + double-click — zoom out 2×.
  • Touch pinch — zoom (mobile / trackpads that support gestures).

2. Static scatterplot — the baseline

Start from a normal scatterplot drawn into the inner g with axes at the edges. Wheel and drag do nothing here — this is the baseline that the next three steps will progressively make navigable:

const xAxisG = plot.append("g")
    .attr("transform", `translate(0, ${innerH})`)
    .call(d3.axisBottom(x));
const yAxisG = plot.append("g").call(d3.axisLeft(y));

plot.selectAll("circle").data(data).join("circle")
    .attr("cx", d => x(d.gdpPercap))
    .attr("cy", d => y(d.lifeExp))
    .attr("r",  d => r(d.pop))
    .attr("fill", d => colour(d.continent));

3. Naïve zoom — transform a wrapping <g>

What changes from §2: we wrap the dots in a new <g> and attach a zoom behaviour that writes the event's transform onto that group. Nothing else moves.

SVG composes the transform with whatever is inside the wrapper, so dots scale and translate as one. Wheel and drag now work — but the axes live outside the wrapper, so they don't move. As you zoom in, dots and axes drift apart and the axis labels stop matching the data:

const zoomG = plot.append("g");                    // ← the wrapper
zoomG.selectAll("circle").data(data).join("circle")
    .attr("cx", d => x(d.gdpPercap))
    .attr("cy", d => y(d.lifeExp))
    .attr("r",  d => r(d.pop));

const zoom = d3.zoom().scaleExtent([1, 16])
    .on("zoom", ({ transform }) => {
        zoomG.attr("transform", transform);        // ← whole group moves
    });
svg.call(zoom);

Three things this code does, in order:

  • plot.append("g") creates the wrapper. All zoomable marks become its descendants. SVG transforms cascade to descendants, so transforming the wrapper transforms everything inside it.
  • svg.call(zoom) attaches the behaviour to the SVG. After this call, wheel / drag / double-click events on the SVG are intercepted by the zoom behaviour and translated into a moving transform.
  • zoomG.attr("transform", transform) — the transform object's toString() produces a valid SVG transform string (translate(120, 80) scale(2.5)), so passing it as the transform attribute applies it directly. Everything inside the wrapper — including the dots' r and any stroke widths — scales with k.

4. Rescale the scales instead

What changes from §3: we drop the wrapper group entirely. The dots go back into plot. The zoom handler now writes data-coordinate changes instead of an SVG transform — it builds rescaled copies of x and y and reapplies them to the dots and the axes.

This is the fix from the canonical Observable example. The transform's rescaleX(x) / rescaleY(y) methods produce rescaled copies of the scales; reapplying those copies to both the dots' cx / cy and the axis generators keeps them in sync — they are computed from the same rescaled scale:

const zoom = d3.zoom().scaleExtent([1, 16])
    .on("zoom", ({ transform }) => {
        const zx = transform.rescaleX(x);          // rescaled x scale
        const zy = transform.rescaleY(y);          // rescaled y scale
        xAxisG.call(d3.axisBottom(zx));            // redraw axes against the new scales
        yAxisG.call(d3.axisLeft(zy));
        dots.attr("cx", d => zx(d.gdpPercap))     // reposition dots
            .attr("cy", d => zy(d.lifeExp));
    });
svg.call(zoom);

Three points worth saying out loud:

  • transform.rescaleX(x) returns a new scale with the same domain as x but a range modified by k and transform.x. The original x is untouched — you can use both, or zero them out by resetting the zoom transform.
  • Why this is better than the group transform. When you rescale the scales, every visual that derives from those scales (axes, gridlines, dots, lines) recomputes against the new ranges and stays consistent. With the group transform you have to remember which elements are inside the wrapper and which are outside.
  • Stroke widths and radii stay constant in screen pixels because the dot's r attribute is unchanged — only cx / cy move. With the group-transform pattern, r would scale with k (a 5-pixel dot becomes 80 pixels at 16× zoom), which is rarely what you want.

Wheel-zoom and drag — the axis ticks now match the dot positions at every level. But: pan to the edge and dots leak outside the plot rectangle, over the axis, into the margins:

5. Clip to the plot rectangle

What changes from §4: the rescale-scales handler is unchanged. We add one ingredient on top — a clipPath declared in the SVG's defs, applied to a new group that wraps the dots. Anything repositioned outside the plot is invisible because SVG only renders the parts of children that fall inside the clip rectangle:

svg.append("defs").append("clipPath").attr("id", "clip")
    .append("rect").attr("width", innerW).attr("height", innerH);

const dotsG = plot.append("g").attr("clip-path", "url(#clip)");
// dots are appended into dotsG instead of plot directly

Two implementation notes:

  • The clipPath rectangle is in plot-coordinates and has a fixed size (innerW × innerH). It does not move with the zoom transform. That is exactly what we want: the clip is a window onto the chart, the scales' rescaling moves the data through that window.
  • The defs element is SVG's way of declaring reusable content (clip paths, markers, gradients) that is rendered only when referenced. The url(#clip) string in clip-path is what activates it.
  • If a page has more than one chart, give each clip-path a unique id (e.g. "clip-" + chartName) — SVG ids share a global namespace, so two charts both using id="clip" will reference the first one and the second chart will look unclipped.

6. Two patterns, two contexts

You have just seen both valid recipes — they are not redundant; they suit different chart families. Rescale-the-scales (§4-§5) is the right default for any chart with axes: scatterplots, line charts, bar charts. Transform the wrapper group (§3) is fine for diagrams without axes — networks, treemaps, abstract layouts — and is what L11 / L12 use for force-directed graphs and chord diagrams.

  • Rescale-scales (axes-aware). Axes follow naturally, stroke widths and mark radii stay constant in screen pixels, the static clipPath works correctly. Use this whenever there are tick labels you want to remain readable.
  • Group-transform (no axes). Simpler code, but everything inside the wrapper scales with k. Combine with vector-effect: non-scaling-stroke on edges to keep edge widths constant under scaling, and counter-scale node radii via r × (1 / k) if you don't want a 5-pixel node to become an 80-pixel blob at 16× zoom.

A practical rule: if your chart has d3.axisBottom / axisLeft anywhere, use rescale-scales. If your chart's data coordinates are abstract (force layout x/y, treemap rectangles), use group-transform.

7. Production checklist

Five questions to ask of any chart in this course before you call it done:

  • Responsive sizing. Set viewBox="0 0 W H" on the SVG and width: 100%; height: auto; in CSS. The chart now scales with the container — the example files in examples/ follow this pattern. For window-resize relayout of force or sankey, listen to resize and rebuild.
  • Accessible colour palettes. The Tableau 10 / Viridis schemes used here are colourblind-aware. For a custom palette, check it through a simulator (e.g. Coblis). Avoid red-green binary contrasts. Don't rely on colour alone — pair colour with shape, position, or label.
  • Accessibility (a11y). SVGs aren't accessible by default. Add <title> children to interactive marks for hover-tooltip text; aria-label on the chart's container summarises the data; consider rendering an equivalent <table> with aria-hidden="true" on the SVG for screen readers. The A11Y Project's SVG guide is the standard reference.
  • SVG vs Canvas. SVG is the default — every example in L9–L12 uses it. For datasets above ~5,000 marks redraw becomes sluggish; switch to <canvas> and draw to its 2D context inside the same tick handler. The data join idea still applies — you just iterate the array and call ctx.beginPath() / arc() / fill() instead of binding DOM elements. Mike Bostock's canvas force-directed graph is the canonical example.
  • Animation honesty. Transitions should explain change, not perform. Default to no transition for high-frequency interactions where the reader expects an immediate response (a brush, a zoom). Use a short transition (300-500 ms) for low-frequency state changes where the eye benefits from following motion: a click that swaps a dataset, a slider that scrubs across discrete years (as in 10c §5 — D3.js interrupts in-flight transitions cleanly when the user keeps dragging). The wrong default is animating every change at 700 ms; that turns an interactive chart into a slideshow.
Takeaway. Navigation is the fifth and final L10 move: change → time → input → composition → navigation. With d3.zoom and the rescale-scales pattern, any chart with axes accommodates more data than fits on one screen. Combined with the four mechanisms from 10a–10d, you have the full advanced D3.js vocabulary.
